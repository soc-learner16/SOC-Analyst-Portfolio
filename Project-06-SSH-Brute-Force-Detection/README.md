# SOC Case 1 — SSH Brute Force Detection & Investigation

## Project Overview

This project demonstrates the complete Security Operations Center (SOC) workflow for detecting and investigating an SSH brute-force attack using Splunk Cloud.

I simulated an SSH brute-force attack from a Kali Linux attacker machine against an Ubuntu Server using Hydra with a controlled custom wordlist. Authentication logs generated by the Ubuntu server (`/var/log/auth.log`) were forwarded to Splunk Cloud using the Splunk Universal Forwarder, where I developed a custom SPL detection rule to identify repeated failed SSH login attempts originating from the same source IP within a two-minute window.

After the detection rule generated an alert, I investigated the underlying authentication events to identify the attacking source IP, determine the targeted user account, reconstruct the attack timeline, and verify whether the attacker successfully authenticated.

This project demonstrates the complete lifecycle of attack simulation, log collection, detection engineering, alert validation, and SOC investigation.

# Attack Scenario

I intentionally used a controlled 15-password wordlist instead of a large public wordlist such as RockYou.

The objective of this project was to demonstrate detection engineering and investigation rather than password cracking. Using a controlled wordlist generated enough authentication events to validate the detection while keeping the dataset clean, repeatable, and easy to investigate.

Hydra generated fourteen failed SSH authentication attempts before successfully authenticating on the final password. This produced both failed authentication telemetry for detection and a successful login event for investigation, closely resembling a real-world brute-force attack.

---

# Detection Logic

SSH brute-force attacks generate numerous authentication failures within a short period of time. Rather than treating every failed login as an individual security event, the detection groups authentication failures by source IP over a two-minute window.

If a single source IP generates five or more failed SSH login attempts during that period, the activity is considered suspicious and an alert is generated.

This approach reduces alert noise by identifying attack patterns instead of individual authentication failures.

Detection conditions:

- Authentication failure recorded in Ubuntu auth.log
- Multiple failures originate from the same source IP
- Five or more failed login attempts occur within two minutes

---

# SPL Detection Query

```spl
index=* host=ubuntuserver sourcetype=auth "Failed password"
| rex "from (?<src_ip>\d+\.\d+\.\d+\.\d+)"
| bin _time span=2m
| stats count by _time src_ip
| where count >= 5
```

### Why this query works

Ubuntu authentication logs store the attacking IP address inside the raw log message rather than as a structured field. Because Splunk does not automatically create a `src_ip` field for the `auth` sourcetype, the IP address must first be extracted using the `rex` command.

Once extracted, authentication events are grouped into two-minute windows using `bin`. The `stats` command counts failed login attempts for each source IP during that period. Any IP producing five or more failures within a single time window is considered suspicious and triggers the alert.

This threshold helps distinguish automated brute-force activity from normal user mistakes, reducing false positives while reliably detecting repeated authentication failures.

# Investigation Process

## 1. Was the alert legitimate?

The triggered alert identified multiple failed SSH authentication attempts originating from a single source IP within the configured detection window, matching the expected behavior of a brute-force attack.

---

## 2. Which host initiated the attack?

Reviewing the authentication logs identified the attacking system as:

**Source IP:** 192.168.10.6

This corresponds to the Kali Linux attacker machine used during the simulation.

---

## 3. Which account was targeted?

Authentication logs showed repeated failed login attempts against the following account:

**Target Username:** kirthi

---

## 4. Did the attacker successfully authenticate?

To determine whether the brute-force attack resulted in account compromise, I searched for successful SSH authentication events.

```spl
index=* host=ubuntuserver sourcetype=auth "Accepted password"
| rex "from (?<src_ip>\d+\.\d+\.\d+\.\d+)"
| table _time src_ip _raw
```

The investigation confirmed one successful SSH authentication after fourteen failed login attempts.

This indicates that the attacker eventually discovered the correct password and established SSH access to the Ubuntu server.

---

## 5. What was the attack timeline?

Authentication events were reviewed chronologically to reconstruct the complete sequence of the attack.

The investigation showed:

- Multiple failed SSH login attempts from 192.168.10.6
- Consistent targeting of the same user account
- Successful authentication after repeated failures

This sequence matches the expected progression of an automated SSH brute-force attack.

# Detection Limitations

This detection identifies repeated failed SSH authentication attempts originating from a single source IP within a two-minute window.

Although effective against traditional brute-force attacks, slower password-spraying attacks or attacks distributed across multiple source IP addresses may not trigger this rule.

In a production SOC environment, this detection would normally be combined with additional correlation rules, behavioral analytics, account lockout monitoring, and threat intelligence enrichment to improve overall detection coverage.

# Analyst Conclusion

The investigation confirmed a successful SSH brute-force attack originating from **192.168.10.6** against the **kirthi** account on the Ubuntu server.

Fourteen failed authentication attempts occurred within a short period before one successful login was recorded. The frequency and consistency of these authentication failures strongly indicate automated password guessing rather than normal user behavior.

The successful authentication confirms that valid credentials were obtained and used to establish SSH access to the target system.

This alert represents a **confirmed True Positive** and should be treated as a compromised account requiring immediate containment.

### Recommended Response Actions

1. Terminate the active SSH session.
2. Block the attacking IP address (192.168.10.6) at the firewall.
3. Reset the compromised user's password.
4. Enable account lockout after repeated authentication failures.
5. Disable SSH password authentication and enforce SSH key-based authentication.
6. Review bash history, sudo logs, and system logs for post-authentication activity.
7. Investigate the possibility of lateral movement or persistence following successful access.
