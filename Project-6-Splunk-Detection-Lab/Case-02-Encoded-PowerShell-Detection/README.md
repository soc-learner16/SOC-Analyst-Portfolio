# SOC Case 2 — Encoded PowerShell Detection & Investigation

## Project Overview

This project demonstrates the detection and investigation of an encoded PowerShell execution using Microsoft Sysmon and Splunk Cloud. PowerShell is a legitimate Windows administration tool, but it is also one of the most frequently abused utilities by attackers because it provides powerful native administrative capabilities while remaining trusted by the operating system.

To reduce the visibility of their activities, attackers frequently execute PowerShell using the `-EncodedCommand` parameter, which accepts Base64-encoded commands instead of plain text. Although the command itself becomes unreadable at a glance, the execution technique leaves identifiable artifacts within endpoint telemetry that defenders can monitor.

In this project, I simulated this technique in a controlled Windows laboratory by executing a Base64-encoded PowerShell command from Command Prompt. Microsoft Sysmon captured the process creation event, which was forwarded to Splunk Cloud through the Splunk Universal Forwarder. I analyzed the generated telemetry, developed a custom SPL detection query, configured a scheduled Splunk alert, and investigated the captured event to determine the executed actions.

The encoded payload was decoded using CyberChef to validate its contents. The decoded commands performed host reconnaissance by retrieving the computer hostname, Windows operating system information, and the current working directory. Although the payload was intentionally benign, the execution technique closely resembles behavior commonly monitored in enterprise environments because attackers frequently abuse encoded PowerShell to conceal command-line activity.

## Attack Scenario

To simulate a commonly monitored PowerShell execution technique, I executed a Base64-encoded PowerShell command from Command Prompt using the `-EncodedCommand` parameter.

The encoded payload contained the following PowerShell commands:

```powershell
hostname;
Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion;
Get-Location
```

The command collected:

- Computer hostname
- Windows operating system information
- Windows version
- Current working directory

The payload was intentionally non-destructive. No persistence, privilege escalation, credential access, or network communication was performed. The objective was to generate realistic endpoint telemetry that could be analyzed and used to build a behavioral detection.

### Why UTF-16LE Encoding?

PowerShell expects the `-EncodedCommand` parameter to receive a UTF-16LE encoded Base64 string rather than a standard UTF-8 Base64 string. During investigation, the payload must therefore be decoded using UTF-16LE to correctly reconstruct the original PowerShell commands.

## Detection Opportunity

The telemetry analysis revealed that the **CommandLine** field consistently contained the `-EncodedCommand` parameter whenever encoded PowerShell was executed.

Unlike the Base64 payload, which changes depending on the executed commands, the execution parameter itself remains consistent. This makes it a reliable behavioral indicator for detection engineering.

Rather than detecting individual PowerShell commands, the detection focuses on identifying the encoded execution technique itself. This provides broader coverage because the rule remains effective regardless of the commands contained within the encoded payload.

---

## Detection Logic

The detection identifies:

- Sysmon Process Creation events.
- PowerShell executions.
- Use of the `-EncodedCommand` or `-enc` parameter.
- User account responsible for execution.
- Parent process.
- Complete encoded command line.

This approach allows analysts to detect encoded PowerShell executions without relying on the decoded payload.

---

## SPL Detection Query

```spl
index=* sourcetype="WinEventLog:Microsoft-Windows-Sysmon/Operational"
EventCode=1
Image="*powershell.exe*"
(CommandLine="*-EncodedCommand*" OR CommandLine="*-enc*")
| table _time Image ParentImage User CommandLine
| sort -_time
```

### Why this query works

| SPL Component | Purpose |
|---------------|---------|
| EventCode=1 | Searches only Sysmon Process Creation events |
| Image="*powershell.exe*" | Restricts results to PowerShell processes |
| CommandLine="*-EncodedCommand*" | Detects the full encoded command parameter |
| CommandLine="*-enc*" | Detects the abbreviated parameter |
| table | Displays investigation-relevant fields |
| sort -_time | Returns the newest events first |

The query was developed directly from the observed telemetry rather than copied from an external source. This reflects a typical detection engineering workflow where endpoint telemetry is analyzed before building a detection rule.

## Detection Limitations

- Detects PowerShell executions using the `-EncodedCommand` or `-enc` parameter.
- Does not detect other PowerShell obfuscation techniques that avoid these parameters.
- Does not automatically decode Base64 payloads.
- Requires Microsoft Sysmon Event ID 1 logging.
- Should be correlated with additional endpoint telemetry before determining malicious intent.

## Analyst Conclusion

This investigation confirmed that Microsoft Sysmon successfully captured the encoded PowerShell execution using Event ID 1 (Process Creation). The generated telemetry was forwarded to Splunk Cloud, where a custom SPL detection query reliably identified the execution based on the presence of the `-EncodedCommand` parameter within the CommandLine field.

Further investigation confirmed that the encoded payload performed host reconnaissance by collecting the computer hostname, Windows operating system information, and current working directory. Although the decoded commands were intentionally benign within this laboratory environment, the execution technique closely resembles behavior frequently monitored in enterprise environments because attackers commonly use encoded PowerShell to obscure command-line activity.
